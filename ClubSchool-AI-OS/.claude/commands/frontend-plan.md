# /frontend-plan

## 목적
퍼블리싱 프로토타입과 화면 요건을 입력으로 **프론트엔드 구현 계획**을 수립한다. 아키텍처·상태관리·라우팅·API 연동·성능 전략을 정의해 개발 착수가 가능한 설계를 만든다.

## 실행 조건
- `/publish-start` 산출물(마크업/컴포넌트)이 존재한다.
- API 명세(또는 백엔드 계획)와 IA 라우팅이 확보되어 있다.

## 호출할 Subagent
- **frontend-lead** (주담당) — FE 아키텍처·구현 계획
- publishing-lead — 마크업/컴포넌트 정합성
- backend-lead — API 계약 정렬
- design-system-lead — 컴포넌트 매핑
- qa-lead — 테스트 전략

## 입력값
- `$ARGUMENTS`: 퍼블리싱 산출물 경로 또는 프로젝트명
- 부가: 기술 스택, API 명세, 성능 예산

## 처리 절차
1. **아키텍처 설계** — 폴더 구조·모듈 경계·렌더링 전략(CSR/SSR)을 정한다.
2. **컴포넌트 매핑** — 디자인시스템 컴포넌트를 코드 컴포넌트로 매핑한다.
3. **상태관리/데이터 흐름** — 전역/지역 상태와 데이터 페칭 전략을 정한다.
4. **라우팅 설계** — IA 라우팅을 라우터 구성으로 변환한다.
5. **API 연동 계약** — 엔드포인트·DTO·에러 처리·로딩 상태를 정의한다.
6. **성능/테스트 전략** — 번들/이미지 최적화, 테스트 범위를 정한다.

## 출력값
```markdown
# 프론트엔드 구현 계획 — <프로젝트명>
## 1. 아키텍처 / 렌더링 전략
## 2. 컴포넌트 매핑 (디자인↔코드)
## 3. 상태관리 / 데이터 흐름
## 4. 라우팅 구성
## 5. API 연동 계약
## 6. 성능 예산 / 테스트 전략
```

## 품질 기준
- [ ] 모든 화면이 라우팅·컴포넌트로 매핑되었다.
- [ ] API 계약이 백엔드와 합의되었다.
- [ ] 상태관리 전략이 화면 복잡도에 적합하다.
- [ ] 성능 예산과 테스트 범위가 정량화되었다.
- [ ] DecisionLog·BestPractices가 갱신되었다.
- 백엔드 정렬이 필요하면 `/backend-plan`과 병행한다.
