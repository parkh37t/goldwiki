# /qa-review

## 목적
임의 단계의 산출물(디자인·퍼블리싱·구현)을 **품질 기준과 테스트 전략**으로 검수한다. 기능·접근성·성능·보안·일관성 관점에서 결함을 적출하고, 심각도와 개선 우선순위를 포함한 QA 리포트를 만든다.

## 실행 조건
- 검수 대상 산출물 또는 구현물이 존재한다.
- 해당 단계의 품질 기준/수용 기준이 정의되어 있다.

## 호출할 Subagent
- **qa-lead** (주담당) — 검수·테스트·결함 관리
- security-risk-lead — 보안 결함 점검
- frontend-lead / backend-lead — 재현/원인 분석
- documentation-lead — 표준 일관성 점검

## 입력값
- `$ARGUMENTS`: 검수 대상 경로 또는 프로젝트명
- 부가: 수용 기준, 지원 환경(브라우저/디바이스)

## 처리 절차
1. **검수 범위 확정** — 대상·환경·수용 기준을 정한다.
2. **기능 검수** — 시나리오/수용 기준 대비 동작을 검증한다.
3. **비기능 검수** — 접근성·성능·반응형·보안을 점검한다.
4. **결함 기록** — 결함을 재현 단계·증거·심각도(Blocker/Critical/Major/Minor)로 기록한다.
5. **회귀 위험 분석** — 변경 영향 범위와 회귀 테스트 항목을 도출한다.
6. **개선 우선순위화** — 심각도×영향으로 수정 우선순위를 정한다.

## 출력값
```markdown
# QA 검수 리포트 — <대상>
## 1. 검수 범위 / 환경
## 2. 결함 목록
| ID | 영역 | 재현 단계 | 기대/실제 | 심각도 | 담당 |
## 3. 접근성/성능/보안 점검 결과
## 4. 회귀 테스트 항목
## 5. 통과/반려 판정 + 개선 우선순위
```

## 품질 기준
- [ ] 모든 수용 기준이 검증되었다.
- [ ] 결함마다 재현 단계와 증거가 있다.
- [ ] Blocker/Critical 결함은 0건이거나 명시적으로 보류 합의되었다.
- [ ] 회귀 위험 항목이 식별되었다.
- [ ] CommonErrors가 갱신되었다.
- 미통과 시 해당 단계 커맨드로 회귀시킨다.
