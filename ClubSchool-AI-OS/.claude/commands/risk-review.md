# /risk-review

## 목적
프로젝트 전반의 **리스크를 체계적으로 식별·평가·대응**한다. 기술·보안·일정·예산·규정·이해관계자 리스크를 영향×발생가능성으로 평가하고, 완화/전가/수용 전략과 담당·트리거를 정한 리스크 레지스터를 만든다.

## 실행 조건
- 분석 대상(RFP/설계/구현 중 하나 이상)이 확보되어 있다.
- 프로젝트 제약(일정·예산·규정)이 식별되어 있다.

## 호출할 Subagent
- **security-risk-lead** (주담당) — 보안·기술 리스크 평가
- pmo-director — 일정·예산·이해관계자 리스크
- backend-lead — 아키텍처 리스크
- cto-reviewer — 기술 타당성 검토
- executive-director — 잔여 리스크 수용 결정

## 입력값
- `$ARGUMENTS`: 프로젝트명 또는 대상 산출물 경로
- 부가: 규정 요건, 제약(일정/예산), 외부 의존성

## 처리 절차
1. **리스크 식별** — 기술·보안·일정·예산·규정·외부의존·이해관계자 카테고리로 도출한다.
2. **평가** — 각 리스크를 영향(1~5)×발생가능성(1~5)으로 점수화한다.
3. **우선순위화** — 리스크 점수로 정렬하고 상위 리스크를 부각한다.
4. **대응 전략** — 완화/전가/회피/수용 중 전략을 선택하고 조치를 정한다.
5. **담당·트리거** — 리스크별 담당과 발현 트리거·조기경보 지표를 지정한다.
6. **컨틴전시** — 발현 시 비상 계획과 예비 자원을 정한다.

## 출력값
```markdown
# 리스크 레지스터 — <프로젝트명>
## 1. 리스크 목록
| ID | 카테고리 | 리스크 | 영향 | 가능성 | 점수 | 전략 | 담당 | 트리거 |
## 2. 상위 리스크 히트맵
## 3. 완화 조치 계획
## 4. 컨틴전시 / 비상 계획
## 5. 잔여 리스크 / 수용 결정
```

## 품질 기준
- [ ] 모든 리스크에 영향×가능성 점수가 있다.
- [ ] 상위 리스크마다 구체 완화 조치와 담당이 있다.
- [ ] 각 리스크에 발현 트리거가 정의되었다.
- [ ] 잔여 리스크에 대한 수용 결정이 기록되었다.
- [ ] DecisionLog·ProjectMemory가 갱신되었다.
- 보안 결함이 발견되면 `/qa-review`와 연계한다.
