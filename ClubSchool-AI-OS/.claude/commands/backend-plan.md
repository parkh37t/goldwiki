# /backend-plan

## 목적
기능 요건·콘텐츠 모델을 입력으로 **백엔드 구현 계획**을 수립한다. 시스템 아키텍처, API 설계, 데이터베이스 스키마, 보안·확장성 전략을 정의해 서버 개발 착수가 가능한 설계를 만든다.

## 실행 조건
- IA의 콘텐츠 모델 또는 요구사항 명세가 확보되어 있다.
- 비기능 요구(성능·가용성·보안)가 식별되어 있다.

## 호출할 Subagent
- **backend-lead** (주담당) — 백엔드 아키텍처·API·DB 설계
- frontend-lead — API 계약 정렬
- security-risk-lead — 인증/인가·보안 설계
- data-analytics-lead — 데이터 모델/적재
- ai-automation-lead — AI/자동화 연동(해당 시)

## 입력값
- `$ARGUMENTS`: IA/요구 산출물 경로 또는 프로젝트명
- 부가: 기술 스택, 비기능 요구(SLA), 통합 대상 시스템

## 처리 절차
1. **아키텍처 설계** — 서비스 경계·계층·배포 토폴로지를 정한다.
2. **데이터 모델링** — ER 다이어그램·스키마·인덱스·정규화 수준을 정한다.
3. **API 설계** — 리소스·엔드포인트·DTO·버전·에러 규격을 REST 표준으로 정의한다.
4. **인증/인가** — 인증 방식·권한 모델·민감정보 처리를 설계한다.
5. **확장성/성능** — 캐싱·큐·트랜잭션·부하 대응 전략을 정한다.
6. **운영/관측성** — 로깅·모니터링·백업·마이그레이션 전략을 정한다.

## 출력값
```markdown
# 백엔드 구현 계획 — <프로젝트명>
## 1. 시스템 아키텍처 (토폴로지)
## 2. 데이터 모델 (ER·스키마·인덱스)
## 3. API 설계 (엔드포인트·DTO·에러)
## 4. 인증/인가 / 보안
## 5. 확장성·성능 전략
## 6. 운영·관측성·백업
```

## 품질 기준
- [ ] 데이터 모델이 IA 콘텐츠 모델과 정합한다.
- [ ] API가 REST 표준과 일관 에러 규격을 따른다.
- [ ] 인증/인가가 최소권한 원칙을 따른다.
- [ ] 비기능 요구(SLA)에 대한 대응 전략이 있다.
- [ ] DecisionLog·BestPractices가 갱신되었다.
- 보안 심층 검토가 필요하면 `/risk-review`로 인계한다.
