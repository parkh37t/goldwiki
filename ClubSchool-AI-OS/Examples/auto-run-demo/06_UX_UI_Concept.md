## 0. 요약 (경영진용 1페이지)

본 문서는 RFP 분석(01)과 IA·유저플로우(05)에서 확정된 **발주처 동인 3축**(① 청소년 70% 모바일 체감, ② KWCAG 2.2 AA 디지털 포용, ③ 행정 효율·데이터 가시성)을 **실행 가능한 모바일 우선 UX 전략·UI 컨셉·접근성 설계**로 번역한 것이다. 평가 항목 "UX/UI 및 접근성 20점"(REQ-013·REQ-014·REQ-021)과 "사업 이해도 20점"의 변별을 직접 겨냥한다.

핵심 결단은 셋이다. 첫째, **단일 디자인 시스템으로 4개 역할(청소년·지도교사/멘토·기관담당·진흥원관리자)을 모두 수용**하되, 청소년 경험은 모바일·터치·경량을, 관리자 경험은 밀도·표·일괄처리를 우선하는 "동일 토큰, 다른 밀도" 전략을 적용한다(REQ-021). 둘째, **접근성을 토큰 단계에서 내장**하여 색 대비·포커스·터치 타깃·320px 리플로우를 디자인 결정의 전제로 강제한다(REQ-013, RISK-07 완화). 셋째, **신뢰가 곧 전환**인 공공 서비스 특성상 증빙 발급(SCR-022)·전자서명(SCR-032)·아동 동의(SCR-003) 화면에 신뢰 신호를 시각 우선순위로 배치한다(H-3, RISK-01·06).

성과 측정은 HEART 프레임워크(UXStrategyFramework §4단계)로 정의하며, 모든 UI 결정은 측정 가능한 가설과 연결한다.

## 1. 모바일 우선 UX 전략 (Discover → Define → Decide → Deliver)

### 1.1 경험 비전 선언문 (Experience Vision Statement)

> "전국 어디의 청소년이든, 저사양 폰과 느린 통신망에서도 **3번의 탭 안에** 동아리에 가입하고 활동을 기록하며, 보호자·지도교사·기관이 **신뢰할 수 있는 증빙과 투명한 절차**로 연결되는 경험."

근거: REQ-014(모바일 우선·청소년 70% 모바일), H-1(모바일 네이티브 수준), H-8(농어촌·저사양·장애 포함 디지털 포용), IA 도달 깊이 ≤ 3.

### 1.2 페르소나 ↔ 핵심 JTBD ↔ 모바일 전략

> RFP §2 4개 역할과 05 IA의 역할 분기를 페르소나로 구체화. 모든 페르소나는 RFP 신호(동인 3축)에 근거하며 가공 인물이 아닌 역할 기반으로 정의한다.

| 페르소나 | 핵심 JTBD(Jobs To Be Done) | 주 사용 맥락 | 모바일 우선 전략 | 근거(REQ/H) |
| --- | --- | --- | --- | --- |
| P1 청소년 회원(주 사용자) | "관심사에 맞는 동아리를 빠르게 찾아 가입하고, 활동을 손쉽게 기록해 생활기록부 증빙을 받고 싶다" | 등하교·쉬는 시간, 모바일, 단속적·저집중 | 하단 탭 5개·엄지 도달 영역·임시저장·경량 번들·오프라인 폴백 | REQ-001·003·004·014, H-1 |
| P2 지도교사/멘토 | "가입·활동을 효율적으로 승인·검토하고 행정 시간을 줄이고 싶다" | 학교·이동 중, 모바일+태블릿 혼용 | 알림 기반 처리·일괄 승인·모바일에서도 검토 가능한 카드 뷰 | REQ-002·003, H-2 |

| P4 진흥원관리자 | "17개 시·도 성과를 한눈에 보고 상위 보고에 활용하고 싶다" | 사무실, PC 중심 | 대시보드·차트·내보내기(데스크톱 최적), 모바일 비대상 | REQ-010, H-2 |

> **역할별 모바일 비중 결정:** 청소년·지도교사 핵심 과업은 모바일 P0(M/T/P 전부), 기관 심사/정산은 T/P, 진흥원 대시보드는 P 전용(05 화면목록 반응형 열과 정합). "모바일 우선"은 전 화면 일률 적용이 아니라 **사용 맥락에 따른 우선순위**로 운영한다.

## 2. UI 컨셉 (톤 · 그리드 · 타이포 · 색 · 핵심 컴포넌트)

> 모든 수치는 디자인 토큰을 단일 출처(SSOT)로 사용한다(UIGuidelines §처리 방식). HEX 하드코딩·임의 px 금지.

### 2.1 톤 & 무드 (Tone of Voice & Visual Tone)

| 차원 | 방향 | 근거 |
| --- | --- | --- |
| 인상 | 신뢰감 있는 공공성 + 청소년 친화적 활기 | 공공 발주(01) + 주 사용자 청소년(P1) |
| 색 온도 | 차분한 블루 계열 기반 + 활동 영역 액센트 | 공공 신뢰 + 영역 식별성 |
| 라이팅 | 명료·고대비, 장식 최소 | KWCAG 2.2 AA(REQ-013), 저사양 성능(H-1) |
| 보이스 | 사용자 용어·친근체("우리 동아리", "활동일지"), 행정 용어 금지 | 05 라벨링 원칙(IA 가이드 §2) |
| 모션 | 기능적·짧게(150–250ms), `prefers-reduced-motion` 존중 | 접근성·저사양 |

### 2.2 그리드 & 레이아웃 (UIGuidelines §그리드 준수)

| 브레이크포인트 | 컬럼 | 거터 | 마진 | 주 사용 화면 |
| --- | --- | --- | --- | --- |
| Mobile (<768px) | 4 | 16px | 16px | 청소년 전 과업(SCR-010~043) |
| Tablet (768–1023) | 8 | 24px | 24px | 지도교사·기관 검토(SCR-060~073) |
| Desktop (≥1024) | 12 | 24px | 컨테이너 max 1200 | 진흥원 대시보드(SCR-080~083) |

- 간격은 4px 베이스 스케일(4/8/12/16/24/32/48/64) **토큰만** 사용 (Don't: 13px·7px 직접 입력)
- 8pt 리듬, 그룹 간 간격 > 그룹 내 간격으로 시각 위계 형성
- 모바일 하단 탭 높이 56px + 안전영역(safe-area-inset) 여백 확보

### 2.3 타이포그래피 (UIGuidelines §타이포 토큰)

| 토큰 | 용도 | 크기/행간 | 굵기 | 적용 예 |
| --- | --- | --- | --- | --- |
| `display` | 히어로 | 40/48 | 700 | SCR-010 홈 환영 영역 |
| `h1` | 페이지 제목 | 32/40 | 700 | 각 화면 타이틀 |
| `h2` | 섹션 | 24/32 | 600 | 카드 그룹·필터 헤더 |
| `body` | 본문 | 16/24 | 400 | 폼·리스트·설명(최소 16px) |
| `caption` | 보조 | 13/18 | 400 | 메타·타임스탬프·도움말 |

- 본문 최소 16px(모바일 입력 시 iOS 자동 확대 방지), 한 줄 45~75자
- 위계는 크기·굵기·색으로만 표현, 브랜드 지정 서체만 사용
- 한글 가독을 위해 줄바꿈은 어절 단위 우선(word-break 보수적 적용)

### 2.4 색상 (시맨틱 토큰 · WCAG 2.2 AA 대비 검증)

> 시맨틱 토큰을 참조하고, 명도 대비를 사전 검증한다. 색만으로 의미 전달 금지(아이콘/텍스트 병행).

| 시맨틱 토큰 | 용도 | 대비 기준 | 색+비색 병행 |
| --- | --- | --- | --- |
| `color-primary` | 주 행동(가입·제출·발급) | 본문 텍스트 ≥ 4.5:1 / UI ≥ 3:1 | 버튼 라벨 텍스트 명시 |
| `color-danger` | 오류·반려·삭제 | ≥ 4.5:1 | 경고 아이콘 + 텍스트 |
| `color-success` | 승인·서명완료·정산완료 | ≥ 4.5:1 | 체크 아이콘 + 텍스트 |
| `color-surface` | 카드·시트 배경 | 전경과 ≥ 3:1 | — |
| `color-text` | 본문 | 배경과 ≥ 4.5:1 | — |

- 영역 액센트(동아리 찾기/우리 동아리/지원금)는 색 + **아이콘 + 라벨**로 구분(색맹 사용자 대응)

## 3. 접근성(KWCAG 2.2 AA) 반영 포인트

> REQ-013 전면 준수, H-8 디지털 포용, RISK-07(미준수 정성 감점) 완화. KWCAG 2.2는 WCAG 2.2 기반이며, 토큰 단계 내장 + CI 자동 점검(axe) + 수동 점검(TC-113)을 병행한다.

### 3.1 KWCAG 2.2 AA 핵심 지침 적용표

| 지침(WCAG/KWCAG) | 적용 규칙 | 적용 화면/컴포넌트 | 검증 |
| --- | --- | --- | --- |
| 1.4.3 명도 대비 | 본문 ≥ 4.5:1, 큰텍스트/UI ≥ 3:1 | 전 화면 토큰 | axe + 대비 측정 |
| 1.4.10 리플로우 | 320px 폭 가로 스크롤 없이 동작 | 전 모바일 화면 | TC-114(3 breakpoint) |
| 1.4.11 비텍스트 대비 | 입력 경계·아이콘 ≥ 3:1 | FormField, 아이콘 | 수동 |

### 3.2 디지털 포용 추가 조치 (H-8 차별화)

- **저대역폭/오프라인 폴백:** 이미지 지연 로드, 핵심 텍스트 우선 렌더, 작성 중 내용 로컬 임시저장(네트워크 단절 대비) — SCR-021·031.

## 6. 품질 체크리스트 (UIGuidelines §체크리스트)

- [x] 모든 간격이 4px 베이스 토큰 스케일을 따른다(임의 px 금지)
- [x] 타이포 위계가 토큰(`display`~`caption`)으로 정의되었다
- [x] 색 대비가 WCAG 2.2 AA(본문 ≥4.5:1, UI ≥3:1)를 충족하도록 토큰 검증 규칙을 명시했다
- [x] 인터랙티브 요소의 6개 상태(Default/Hover/Focus/Active/Disabled/Loading)를 정의했다
- [x] Empty/Error/Loading 화면 상태를 정의했다(§2.5, 05 §3.2와 정합)
- [x] 터치 타깃 ≥ 44px, 320px 리플로우 OK(1.4.10·2.5.8)
- [x] 색만으로 의미 전달하지 않는다(아이콘/텍스트 병행)
- [x] 포커스 링을 명시했다(제거 금지, 2.4.7·2.4.11)
- [x] WCAG 2.2 신규 항목(2.4.11·2.5.7·2.5.8·3.3.7·3.3.8) 적용 규칙을 명시했다
- [x] 모든 UI 결정이 REQ↔SCR↔TC로 추적 가능하다(§5 RTM)

**Do / Don't (UIGuidelines §품질 기준)**

| Do | Don't |
| --- | --- |
| 4px 토큰 간격·시맨틱 색 토큰 참조 | 임의 px·HEX 하드코딩 |

- **QA:** §5 RTM의 UI↔TC 연결 → TC-101·104·105·106·110·113·114·119·121 점검 시드.

**거버넌스 갱신 트리거(의사결정 로그·프로젝트 메모리 기록 대상):**
- "동일 토큰·다른 밀도" 역할별 UI 전략 채택(REQ-021, 단일 디자인 시스템 4역할 수용)
