## 0. 개요와 설계 동인

본 문서는 RFP 분석(01)에서 확정된 22개 요구를 **구현 가능한 아키텍처·기술스택·API·데이터·보안 설계**로 변환한다. 설계의 4대 동인은 다음과 같으며, 모든 설계 결정은 이 동인과 골드위키 표준에 정렬한다.

| 동인 | 근거(REQ/RISK) | 본 문서에서의 설계 귀결 |

## 1. 시스템 아키텍처 (공공 클라우드 · 행정망 연계)

### 1.1 배치 토폴로지 (2-Zone 망분리)

청소년·지도교사·기관담당은 **인터넷 구간(공공 클라우드 IaaS/PaaS)** 으로 접속하고, 진흥원 관리자 백오피스와 행정 연계(본인확인·정산·행정망 API)는 **행정망 구간**에서 동작한다. 두 구간은 단방향에 가까운 통제된 연계 게이트웨이(연계 서버 + 큐)로만 연결한다(REQ-018).

```
[ 인터넷 사용자 ]  청소년/지도교사/기관담당  (모바일 70%+)
        │  HTTPS/TLS1.3
        ▼
┌──────────────────────── 인터넷 Zone (공공 클라우드) ────────────────────────┐
│  CDN(정적·이미지·증빙PDF 캐시)  →  WAF  →  L7 LB                              │
│        │                                                                     │
│   대기열 게이트(Virtual Waiting Room) ── 피크 시에만 활성 (REQ-019)          │
│        ▼                                                                     │
│   API Gateway (인증·레이트리밋·라우팅)                                       │
│   ├── BFF (프론트 전용 집계 API, SSR/RSC)                                    │
│   ├── 회원/인증 서비스   ├── 동아리/활동 서비스   ├── 지원금/정산 서비스     │
│   ├── 알림 서비스        ├── 증빙발급 서비스      └── 통계/대시보드 리드모델 │
│        │                         │                          │               │
│   [ PostgreSQL Primary+Read Replica ]  [ Redis(캐시·큐·세션) ]  [ Object 스토리지 ] │
│        │                         │                                          │
│   [ 메시지 큐 / 워커(BullMQ·DLQ) ]  ← 비동기·피크 흡수                       │
└──────────────────────────────┬──────────────────────────────────────────────┘
                                │  연계 큐 + 연계 서버 (서명·화이트리스트·감사)
┌──────────────────────────────▼──────────────────────────────────────────────┐
│  행정망 Zone                                                                  │
│   ├── 본인확인/실명·연령확인 연계(REQ-009)   ├── 행정 정산·회계 연계(REQ-022) │
│   ├── 진흥원 관리자 백오피스(SCR-070대)       └── 감사·로그 보존(변조방지)     │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 아키텍처 결정 (ADR 후보 — 의사결정 로그 기록 대상)

| ID | 결정 | 대안 | 채택 근거 | 연결 |
| --- | --- | --- | --- | --- |
| AD-01 | **모듈러 모놀리스 + 도메인 분리**로 출발, 피크 도메인(지원금)만 독립 배포 | 풀 MSA / 단일 모놀리스 | 9개월 고정 일정(RISK-04)에 MSA 운영 부담 과다. 헥사고날 경계([21 §2])로 향후 분리 여지 확보 | REQ-018, REQ-020 |
| AD-02 | **외부 연계는 어댑터 패턴 + 연계 큐**로 격리, mock 우선 개발 | 직접 동기 호출 | 행정망 규격·일정 지연(RISK-05) 흡수, 인터넷↔행정망 결합도 최소화 | REQ-007/009/018 |
| AD-03 | **읽기 전용 리드 모델(구체화 뷰/캐시)로 대시보드·통계 분리** | 운영 DB 직접 집계 | 마감일 피크 시 통계 쿼리가 트랜잭션 경로를 침범하지 않게 분리([23 §9]) | REQ-010, REQ-019 |
| AD-04 | **상태 무관(Stateless) 서비스 + 외부 세션 저장(Redis)** | 서버 세션 고정 | 수평 오토스케일 전제([21 §1]) | REQ-019 |

원칙: 도메인 계층은 프레임워크·DB·행정망 규격을 알지 못한다(의존성은 안쪽으로, [21 §2]). 모든 외부 연계는 포트-어댑터로 교체 가능하게 한다.

## 2. 기술 스택과 채택 근거

골드위키 표준 스택([20 §1], [21])을 본 사업 제약(공공 클라우드·모바일 70%·접근성·피크)에 맞춰 확정한다.

### 2.1 프론트엔드 ([20 §1] 정합)

| 영역 | 채택 | 근거(REQ 연결) |
| --- | --- | --- |
| 프레임워크 | **Next.js (App Router) + TypeScript** | 모바일 LCP·SEO·SSR/RSC로 초기 JS 최소화(REQ-014), 청소년 저사양 단말 대응(H-8) |
| 빌드 | Next.js 내장 + Turbopack/webpack | 표준([20 §1]) |
| 스타일 | **CSS 모듈 + 디자인 토큰 변수** | 디자인 시스템 재사용·런타임 비용 0, 하드코딩 금지([18], [20 §5]) |
| 서버 상태 | **TanStack Query** | 캐싱·재검증·중복 제거, 피크 시 클라이언트 재시도 백오프(REQ-019) |
| 클라이언트 상태 | **Zustand** | 경량 전역(테마·역할 컨텍스트·대기열 상태) |
| 폼·검증 | **React Hook Form + Zod** | 지원금 신청 다단계 폼·서버 스키마 공유(REQ-005) |
| 접근성 | eslint-plugin-jsx-a11y, axe-core, Lighthouse CI | KWCAG 2.2 AA 자동·수동 점검(REQ-013, [20 §7]) |
| 테스트 | Vitest + Testing Library + Playwright | [20 §8], [30] 정합 |

> **모바일 우선 보강:** 라우트 단위 코드 분할, LCP 이미지 `preload`+`fetchpriority`, `font-display: swap`+woff2 서브셋, 이미지 AVIF/WebP `srcset`. 저대역폭 폴백(스켈레톤·오프라인 안내)으로 디지털 포용(H-8)을 차별화한다. 성능 예산은 §5.4.

### 2.2 백엔드 ([21], [22], [23] 정합)

| 영역 | 채택 | 근거 |
| --- | --- | --- |
| 런타임/언어 | **Node.js + TypeScript (NestJS 권장)** | 프론트와 Zod 스키마·타입 공유, 헥사고날 구조 표현 용이([21 §2~3]) |
| 아키텍처 | **헥사고날(포트-어댑터) + 모듈러 모놀리스** | AD-01 |
| 데이터베이스 | **PostgreSQL 15+ (Primary + Read Replica)** | 제약·트랜잭션·트라이그램 검색·PITR([23]) |
| 캐시·큐·세션 | **Redis** + **BullMQ(워커·DLQ)** | cache-aside, 대기열, 비동기 정산·알림([21 §8~9]) |
| 오브젝트 스토리지 | 공공 클라우드 S3 호환 | 활동사진·첨부서류·증빙PDF, CDN 오리진 |
| 관측성 | OpenTelemetry + Prometheus + 구조화 JSON 로그(Loki/ELK) | 골든 시그널 알림([21 §5]) |
| 인증 | OAuth2/OIDC, 단기 JWT + 리프레시 회전 | [21 §7], [22 §8], [24 §3] |
| 검색 | PostgreSQL `pg_trgm`(초기) → 트래픽 증가 시 검색엔진 분리 | 동아리 검색(REQ-001), 과설계 회피 |

> 스택 변경은 ADR로 [의사결정 로그](../../GoldWiki/32_DECISION_LOG.md)에 기록한다([20 §1]). 공공 클라우드 PaaS 제공 관리형 PostgreSQL·Redis를 우선 사용해 운영 부담을 낮춘다.

## 3. API 설계 원칙

[22 · API 표준]을 전면 채택한다. 본 사업 특화 규약만 아래에 명시한다.

### 3.1 표준 준수 요약 (체크리스트)

- [x] **리소스 중심·복수 명사·소문자**: `/v1/clubs`, `/v1/grant-applications` ([22 §2])
- [x] **URL 경로 메이저 버전** `/v1` ([22 §4])
- [x] **표준 에러 envelope** + `request_id` ([22 §6], [21 §6]와 정합)
- [x] **커서 페이지네이션** 기본(대용량 동아리·활동 목록) ([22 §5])
- [x] **OAuth2/JWT + 스코프** 최소 권한(`clubs:read`, `grants:write` 등) ([22 §8])
- [x] **레이트리밋** `RateLimit-*`/`Retry-After`, 키 단위(사용자/IP) ([22 §9])
- [x] **멱등성 키** 비멱등 POST(지원금 신청·정산)에 `Idempotency-Key` ([22 §7])
- [x] **OpenAPI 3.1 계약 우선** — 코드·목·계약테스트 생성 ([22 §1], [30])
- [x] **웹훅** 행정 연계 결과 수신은 HMAC-SHA256 서명·멱등 처리 ([22 §11])

### 3.3 사업 특화 규약

```
# 동아리 검색 — 커서 페이지네이션·필터·정렬 ([22 §5])
GET /v1/clubs?region=seoul&interest=music&age=15&day=sat
            &sort=-created_at&page[size]=20&page[cursor]=eyJpZCI6...

# 지원금 신청 — 멱등성 필수(중복 제출 방지) ([22 §7])
POST /v1/grant-applications
Authorization: Bearer <jwt; scope=grants:write>
Idempotency-Key: 5f3c2b1a-...        # 마감일 더블클릭/재시도 중복 차단(REQ-019)

# 피크 대기열 응답 — 입장 전 토큰 발급 (§5.2)
HTTP/1.1 202 Accepted
Retry-After: 30
{ "queue": { "position": 1840, "token": "wr_01H...", "eta_seconds": 90 } }
```

```jsonc
// 행정 연계 결과 웹훅 수신 — 서명·멱등 ([22 §11])
POST /v1/webhooks/admin-net
X-GW-Event: settlement.confirmed
X-GW-Signature: sha256=...           // HMAC 검증 후 처리, id로 중복 무시
{ "id": "evt_01H...", "type": "settlement.confirmed", "data": { "application_id": "ga_1" } }
```

> 모든 상태변경(가입승인·심사·정산)은 POST 하위 액션으로 표현하고([22 §2]), 응답·에러는 표준 envelope로 통일한다. 본인확인·행정망 연계는 **동기 호출 금지**, 연계 큐를 통해 비동기 처리하여 피크·지연을 흡수한다(AD-02).

## 4. 데이터 이관 방안 (회원 30만 · 동아리 1.2만)

REQ-016 / RISK-03(매우높음). 목표: **무손실·정합률 100%·롤백 가능·병행운영**. [23 §6 마이그레이션]의 확장-수축 패턴을 데이터 이관에 확장 적용한다.

### 4.1 이관 대상·규모·난이도

| 데이터셋 | 건수 | 난이도 | 핵심 리스크 |
| --- | --- | --- | --- |
| 회원(개인정보·인증·연령) | 약 300,000 | 상 | 중복·결측 식별자, 아동 동의 이력 부재, 암호화 필드 |
| 동아리(기본·소속·담당) | 약 12,000 | 중 | 기관 코드 매핑, 중복 동아리 통합 |
| 회원-동아리 관계(소속) | 30만~수십만 | 상 | N:M 참조 무결성 |
| 활동기록·증빙(이력) | 가변 | 중 | 첨부파일 누락, 기간 정합 |
| 지원금·정산 이력 | 가변 | 상 | 금액 합계 정합, 회계 추적성(REQ-022) |

### 4.2 6단계 이관 절차 (FLOW-06)

```
1. 프로파일링/매핑정의  →  2. 스테이징 적재(ETL)  →  3. 정제·표준화·중복제거
        →  4. 단계 적재(배치, CONCURRENTLY)  →  5. 정합률 검증(자동)  →  6. 컷오버/롤백
```

| 단계 | 활동 | 산출물 | 검증(TC) |
| --- | --- | --- | --- |
| 1 프로파일링 | 소스 스키마·품질·결측·중복 분석, **필드 매핑 정의서** 작성 | 매핑정의서·데이터 사전 | — |
| 2 스테이징 | 운영과 분리된 스테이징 DB로 무변환 적재(원본 보존) | 스테이징 적재 로그 | TC-116 |
| 3 정제 | 표준화(연락처·지역코드), 중복 통합, 아동 동의 결측 플래그 | 정제 규칙·예외 리스트 | TC-116 |
| 4 적재 | 배치 분할 적재, `CREATE INDEX CONCURRENTLY`로 락 최소화([23 §6]) | 적재 리포트 | TC-116 |
| 5 검증 | **건수·합계·체크섬·표본 대조** 정합률 측정, 100% 미만 시 차단 | 정합률 리포트 | TC-116 |
| 6 컷오버 | 읽기 동결 → 증분 동기화 → 전환 → 스모크, 실패 시 롤백 | 컷오버 런북·롤백 로그 | TC-116, TC-120 |

### 4.3 정합률 검증 쿼리(예시) — [23 §8] 정합

```sql
-- (1) 건수 정합: 소스 vs 타깃
SELECT 'members' AS dataset,
       (SELECT count(*) FROM staging.members)        AS source_cnt,
       (SELECT count(*) FROM users WHERE deleted_at IS NULL) AS target_cnt;

-- (2) 금액 합계 정합(지원금·정산 회계 추적성, REQ-022)
SELECT sum(amount) AS source_sum FROM staging.settlements
EXCEPT
SELECT sum(amount) AS target_sum FROM settlements;   -- 차집합 비면 일치

-- (3) 참조 무결성: 고아 관계 0건이어야 함
SELECT count(*) AS orphan_memberships
FROM memberships m
LEFT JOIN users u ON u.id = m.user_id
WHERE u.id IS NULL;

-- (4) 표본 필드 대조(해시 비교)
SELECT s.legacy_id
FROM staging.members s
JOIN member_id_map x ON x.legacy_id = s.legacy_id
JOIN users u ON u.id = x.new_id
WHERE md5(s.full_name||s.birth_ym) <> u.profile_hash;   -- 불일치 표본
```

### 4.4 매핑·전략 (확장-수축 + 병행운영)

```sql
-- 레거시 ID ↔ 신규 UUID 매핑 테이블(롤백·증분 동기화 기준점)
CREATE TABLE member_id_map (
  legacy_id   TEXT PRIMARY KEY,
  new_id      UUID NOT NULL REFERENCES users(id),
  migrated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- **병행 운영 기간 확보**: 신규 시스템 오픈 후 일정 기간 레거시를 읽기 전용으로 유지, 증분 변경분만 재동기화(RISK-03·H-6, Q&A 후보 2 해소 전제).
- **컷오버 롤백**: 컷오버 실패 시 매핑 테이블 기준으로 즉시 레거시 복귀. RPO ≤ 15분/RTO ≤ 1시간 목표([23 §10]).
- **아동 동의 결측 처리**: 만 14세 미만 동의 이력이 없는 회원은 이관 후 **재동의 유도 플래그**로 격리, 미동의 시 기능 제한(REQ-008, RISK-01).
- **암호화 필드**: 주민/연락처 등 민감 식별자는 이관 시점에 컬럼 단위 암호화·토큰화 적용(§6, [24 §6], [23 §1·§11]).

> 이관은 단순 적재가 아니라 **검증·롤백·병행운영을 포함한 프로젝트**로 다룬다. 정합률 100% 검증 통과 전에는 컷오버를 금지한다(게이트).

## 5. 피크 대응 (지원금 마감일 동시접속)

REQ-019 / RISK-02(높음). 마감 직전 수 시간에 신청·첨부 업로드·심사 조회가 폭증한다. 전략: **유입 평탄화 → 읽기 캐싱 → 쓰기 비동기 → 자원 확장**의 4중 방어.

### 5.1 부하 가정·목표

| 항목 | 가정/목표 | 근거 |
| --- | --- | --- |
| 마감일 피크 동시접속 | 설계 목표 상한 산정 후 1.5배 버퍼로 부하 테스트 | RFP_INPUT §5, H-5 |
| 신청 제출 응답 | p95 < 2초, 실패율 < 0.5% | NF 요구·관행 |
| 가용성 | 마감 윈도 99.9% | RISK-02 완화 |
| 부하 테스트 | 마감일 가정 시나리오 **조기 수행**(단계 17 이전) | RISK-02 대응 |

### 5.2 가상 대기열 (유입 평탄화)

```
[사용자] → 대기열 게이트 → (정원 미만) 입장 토큰 발급 → API Gateway
                         → (정원 초과) 202 + position/eta, 클라이언트 폴링(Retry-After)
```

- 마감일 등 지정 윈도에만 활성화. 입장 정원은 백엔드 처리 용량 기준 동적 조절.
- 입장 토큰은 단기·서명 검증. 신청 제출은 §3.3의 `Idempotency-Key`로 중복 제출을 흡수(REQ-019).
- 대기 UX는 SCR-030대 신청 진입 화면에 위치, 접근성 준수(REQ-013).

### 5.3 캐싱·CDN·비동기 (계층 방어)

| 계층 | 기법 | 비고 |
| --- | --- | --- |
| CDN | 공고·안내·증빙PDF·정적 자산 엣지 캐시 | 마감일 조회 트래픽의 대부분을 오리진 이전에서 흡수 |
| HTTP 캐시 | `Cache-Control`/`ETag` 조건부(304) | [21 §8] |
| 앱 캐시 | Redis cache-aside, 공고·자격 조회 TTL | [21 §8] |
| 리드 모델 | 대시보드·통계 구체화 뷰 | 통계가 트랜잭션 경로 침범 방지(AD-03) |
| 쓰기 평탄화 | 신청 접수=`202` 후 큐 적재, 워커가 심사·정산 처리 | [21 §9], BullMQ 멱등·DLQ |
| DB | 읽기 복제본 분산, PgBouncer 풀링, 부분/복합 인덱스 | [23 §5·§9] |

```ts
// 신청 접수: 빠른 응답 + 비동기 처리 ([21 §9], [22 §3])
async function submitGrantApplication(input, idemKey) {
  const existing = await idem.get(idemKey);        // 멱등 캐시(24h)
  if (existing) return existing;                    // 중복 제출 → 동일 결과
  const id = await repo.insertPending(input);       // 짧은 트랜잭션만([23 §7])
  await grantQueue.add("review", { id }, {          // 무거운 처리는 워커로
    attempts: 3, backoff: { type: "exponential", delay: 2000 }, removeOnComplete: 1000,
  });
  const res = { status: 202, application_id: id };
  await idem.set(idemKey, res, 86400);
  return res;
}
```

### 5.4 자원 확장·성능 예산

- **오토스케일**: stateless 서비스·워커를 CPU/큐 길이 기준 수평 확장(AD-04). DB는 읽기 복제본 추가로 조회 확장.
- **프론트 성능 예산**([20 §6]): LCP < 2.5s, INP < 200ms, CLS < 0.1, 초기 JS gzip < 170KB. CI `size-limit`·Lighthouse CI로 강제(TC-114, [20 §8]).
- **카오스·부하 테스트**(TC-119): 마감일 1.5배 시나리오, 큐 포화·DB 페일오버·연계 지연 주입 검증.
