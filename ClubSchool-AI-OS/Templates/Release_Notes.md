# 릴리스 노트 템플릿 (Release Notes)

> **용도**: 특정 버전에서 무엇이 추가·변경·수정됐는지를 이해관계자(고객·운영·개발)에게 전달하는 배포 공지 문서.
> **사용 에이전트**: DevOps Engineer(주), Documentation Specialist, Product Owner.
> **연계 산출물**: [`Test_Plan.md`](Test_Plan.md)(검증 결과), [`WBS.md`](WBS.md)(마일스톤).
> **관련 GoldWiki**: [31 릴리스 프로세스](../GoldWiki/31_RELEASE_PROCESS.md)

### 작성 팁
- **버전은 SemVer로**: `MAJOR.MINOR.PATCH`. 호환성 깨짐=MAJOR, 기능 추가=MINOR, 버그 수정=PATCH.
- **사용자 관점으로 쓴다**: 내부 구현이 아니라 "사용자에게 무엇이 달라지는가"를 먼저 말한다.
- **Breaking Change는 최상단에**: 마이그레이션 영향이 있으면 가장 먼저, 굵게 표시한다.
- **추적 가능하게**: 각 항목에 이슈/PR/티켓 번호를 연결한다.
